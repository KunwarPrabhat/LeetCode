# DSA Problem Solving Strategy (LeetCode Focus)

## Goal
Build strong problem-solving skills for technical interviews by focusing on:
- Pattern recognition
- Speed and accuracy
- Deep understanding instead of shallow volume

---

## Core Philosophy

Solving more problems does not guarantee improvement.  
Recognizing patterns, understanding mistakes, and revisiting problems leads to real progress.

---

## Problem Solving Strategy

### 1. Daily Problem Volume
- 8–12 problems per day
  - 2–3 Easy (warm-up)
  - 5–7 Medium (primary focus)
  - 0–2 Hard (optional)

---

### 2. Time Limits per Problem
- Easy: 10–15 minutes  
- Medium: 30–40 minutes  
- Hard: 60 minutes  

If unable to solve within the time limit:
- Stop
- Read the optimal solution
- Understand the approach fully
- Re-implement without looking

---

### 3. Solve Process (Per Problem)

1. Read the problem carefully  
2. Identify constraints and edge cases  
3. Attempt solution within time limit  
4. If stuck:
   - Analyze why
   - Study the correct approach
5. Re-code the solution from scratch  
6. Note the pattern or technique used  

---

### 4. Pattern Tracking

For each problem, identify:
- Technique used (e.g., sliding window, binary search, DFS)
- Key insight or trick
- Mistakes made

Focus on recognizing patterns rather than memorizing problems.

---

### 5. Revision Strategy

- Re-solve problems after 2–3 days
- Re-solve again after 1 week
- If unable to solve again, treat it as not learned

Revision is critical for retention and speed.

---

### 6. Topic Focus (Generic Order)

- Arrays and Hashing  
- Strings  
- Two Pointers  
- Sliding Window  
- Binary Search  
- Stack and Queue  
- Trees (DFS, BFS)  
- Recursion and Backtracking  
- Graph Basics  
- Dynamic Programming (basic to intermediate)

---

### 7. Weak Area Handling

If repeatedly struggling with a topic:
- Pause new problems in that area
- Study core concepts
- Solve 3–5 focused problems
- Resume mixed practice

---

### 8. Common Mistakes to Avoid

- Spending too long stuck on one problem  
- Watching solutions without re-implementing  
- Not revisiting solved problems  
- Focusing only on easy problems  
- Ignoring edge cases  

---

### 9. Progress Indicators

You are improving if:
- Medium problems feel more familiar  
- You recognize patterns faster  
- You make fewer implementation mistakes  
- You solve previously solved problems faster  

---

## Final Note

The goal is not to complete problems, but to internalize problem-solving patterns.
